# Model Support Matrix

This document summarizes the current support status, missing components, and future integration strategies for the target models and modes.

| Model / Feature | Current Status | Missing Components | Future Strategy |
| :--- | :--- | :--- | :--- |
| **Nemotron-Labs-Diffusion-3B (Diffusion Mode)** | **Partially supported** (Definition only) | • The [Scheduler](file:///Users/yugeshk/dev/repo/omlx/omlx/scheduler.py) is tightly coupled to `BatchGenerator` and causal autoregressive decoding.<br>• `DiffusionStrategy` in [diffusion.py](file:///Users/yugeshk/dev/repo/omlx/omlx/inference/strategies/diffusion.py) is a non-functional stub.<br>• `ExperimentalNemotronBackend` in [experimental_diffusion_backend.py](file:///Users/yugeshk/dev/repo/omlx/omlx/inference/backends/experimental_diffusion_backend.py) has a mock execution pipeline that doesn't bind to actual loaded models. | • Refactor the Scheduler to delegate step execution to `BaseGenerationStrategy.forward()`.<br>• Fully implement `DiffusionStrategy` and bind it to the actual loaded weights via `NemotronModelAdapter` in [nemotron_adapter.py](file:///Users/yugeshk/dev/repo/omlx/omlx/models/adapters/nemotron_adapter.py). |
| **Diffusion Gemma** | **Partially supported** (Serial bypass only) | • Classified as `vlm` engine type in [model_discovery.py:L600](file:///Users/yugeshk/dev/repo/omlx/omlx/model_discovery.py#L600) and run under `VLMBatchedEngine` in [vlm.py:L2801](file:///Users/yugeshk/dev/repo/omlx/omlx/engine/vlm.py#L2801).<br>• Bypasses the scheduler entirely and runs sequentially under a single-request lock (`self._diffusion_lock` in [vlm.py:L3710](file:///Users/yugeshk/dev/repo/omlx/omlx/engine/vlm.py#L3710)), preventing concurrency and multi-request scheduling. | • Migrate `mlx-vlm`'s block-diffusion execution logic from the thread-pool bypass to the unified `DiffusionStrategy` and `DiffusionBackend` execution paths.<br>• Enable continuous scheduler dispatch for block-diffusion steps. |
| **Nemotron Labs 3B Triage Mode** | **Unsupported** (Not in codebase) | • No capability flags (`supports_triage`, `supports_verification`) in [capabilities.py](file:///Users/yugeshk/dev/repo/omlx/omlx/runtime/capabilities.py).<br>• No strategy or backend class for hybrid self-speculation (which alternates between diffusion drafting and AR verification). | • Extend capability resolution and profile registries.<br>• Implement a hybrid `TriageStrategy` that constructs a multi-stage `ExecutionGraph` (Prefill -> Draft -> Verify -> Accept -> Emit) and coordinates dynamic masking in the adapter. |
| **Streaming MoE execution** | **Unsupported** (Structural stubs only) | • Current MoE model loading uses static patching ([omlx/patches/deepseek_v4](file:///Users/yugeshk/dev/repo/omlx/omlx/patches/deepseek_v4) and [omlx/patches/glm_moe_dsa](file:///Users/yugeshk/dev/repo/omlx/omlx/patches/glm_moe_dsa)) without stream-aware expert allocation.<br>• No strategy/backend class for dynamic expert activation or offloading. | • Implement `StreamingMoeStrategy` and a streaming-capable execution backend.<br>• Leverage `CacheManager` and `ProcessMemoryEnforcer` to page/offload experts dynamically during prefill to manage peak memory. |
